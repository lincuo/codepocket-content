# 每日架构原则｜2026-07-24

## 设计原则不是圣经——在嵌入式世界里，它们是有条件的启发式

PC 后端写惯了的人，最容易犯的一个错误是把 SOLID、DRY、KISS 当作放之四海而皆准的教条。这些原则诞生于 Java 企业级应用和大型代码库的土壤里——那里有 GC 兜底、有充足内存、有热部署可以回滚。嵌入式系统不共享这些前提条件。

**KISS 在嵌入式里不是"写最简单的代码"，而是"接受不完美的简单"。** 一个用全局变量直接传递传感器读数的裸机循环，比一个精心设计的观察者模式主题/订阅器更可靠——因为后者引入了你未必需要的事件分发层、回调链和生命周期管理。嵌入式系统的复杂度守恒定律同样适用：你只是把运行时耦合换成了编译时耦合。

**DRY 在嵌入式里有物理代价。** 重复一段 15 行的 UART 初始化代码四次，占 60 字节 Flash。把它抽成通用函数，加上参数抽象和泛型处理，可能占 200 字节 Flash 加上一段间接跳转开销。在 Flash 只有 256KB 的 MCU 上，这 140 字节的差额意味着你能多存一个完整的 OTA 校验表，或者少一次外部 SPI Flash 读取。

**SOLID 里的依赖倒置原则（DIP）在嵌入式里需要重新理解。** 你说"领域层不应该依赖 HAL"——对，但你的领域层最终要调用 `HAL_GPIO_WritePin()`。真正的做法不是消除依赖，而是通过一个薄弱的硬件抽象接口（比如 `led_toggle()`）来隔离变化。当你要从 STM32 换到 nRF52 时，你只需要修改这个接口的实现，而不是翻遍整个状态机逻辑。

### 反模式示例：FreeRTOS 任务中的"过度抽象"

```c
/* ❌ 一个传感器读数，搞出三层接口 */
typedef struct Sensor Sensor_t;
struct Sensor {
    const SensorOps_t *ops;
    void *ctx;
};

typedef struct {
    int32_t (*read)(Sensor_t *self);
    void (*init)(Sensor_t *self);
} SensorOps_t;

/* 调用方：为了读一个温度值，经过三次间接跳转 */
int32_t temp = sensor->ops->read(sensor);

/* ✅ 直接调用的接口，零抽象开销 */
int32_t bme280_read_temperature(BME280_Handle_t *h);
```

这段反模式的代价是什么？

1. **Flash**：`SensorOps_t` 虚表 + 每个实例的指针 = 每传感器 8 字节 RAM + 虚表代码段
2. **性能**：三次间接跳转（指针解引用 → 函数指针解引用 → 目标函数），在 Cortex-M 上至少 6 个周期
3. **调试**：断点打在 `read()` 上，你不知道是哪个传感器的哪个实现在执行

这不是说接口抽象不好——好的抽象是在模块边界（I2C 驱动层），不是在数据通路的核心路径上。

### 实践建议

**给你的每个设计原则加一个"嵌入式条件检查"：**

| 原则 | PC 后端的判断标准 | 嵌入式的额外检查 |
|------|-------------------|-----------------|
| KISS | 代码是否易懂？ | 抽象层引入的间接跳转是否在关键路径上？ |
| DRY | 重复是否超过两次？ | 合并后的代码增量是否超过 50 字节 Flash？ |
| DIP | 高层不应依赖低层？ | 接口隔离的成本是否低于切换芯片时的修改量？ |

具体操作：在你的项目根目录建一个 `ARCHITECTURE-DECISIONS.md`，每次决定使用某个设计原则时，写下：

```markdown
## ADR-003: 传感器读数不使用虚表抽象

**决策**：BME280 驱动直接暴露 `bme280_read_temperature()` 函数，
不通过 Sensor 接口抽象。

**理由**：
- 传感器读数在关键控制回路中，零间接跳转是硬性要求
- 当前项目只使用 BME280，切换成本为零
- 虚表方案增加 8 字节 RAM + 6 周期延迟，收益为零

**何时 reconsider**：当项目需要同时支持 3+ 种温度传感器
且它们的读数接口存在显著差异时。
```

这种 ADR 记录的不是"我们用了什么"，而是"我们为什么拒绝了另一种方案"——这才是架构决策的真正价值。

**一句话总结：嵌入式的设计原则不是被抛弃，而是被重新加权——可预测性永远高于优雅，确定性永远高于灵活。**

> 《代码大全》第 2 版："抽象应该揭示相似性，而不是隐藏差异性。当一个抽象开始掩盖重要的行为差异时，它就不再是抽象，而是欺骗。"

**今日思考题**：打开你当前项目的某个驱动模块，数一下从 API 调用入口到最终硬件寄存器写入之间，经过了几个函数间接跳转。如果这个数字大于 3，问自己：这个抽象在什么场景下会被替换掉？如果答案是不太可能，考虑简化它。
