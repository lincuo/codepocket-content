# 每日架构原则 — 圈复杂度与代码度量在资源受限环境下的意义

## 度量不是尺子，是X光片

圈复杂度（Cyclomatic Complexity）和耦合度是PC后端最常用的静态分析指标，但在嵌入式固件里，它们经常给出误导性的"健康"信号。

问题出在哪？

一个STM32 FreeRTOS任务里，你写了一个状态机：

```c
/* ❌ 反模式：低圈复杂度，高耦合 */
void task_sensor(void *arg) {
    while(1) {
        switch(get_state()) {
            case IDLE:   read_adc(); break;
            case WORK:   calc_filter(); break;
            case ERROR:  log_error(); break;
        }
        vTaskDelay(pdMS_TO_TICKS(10));
    }
}

/* 全局变量：状态、ADC原始值、滤波系数、错误计数 */
extern uint8_t g_state;
extern int32_t g_adc_raw;
extern float g_coeff;
extern uint32_t g_err_cnt;
```

圈复杂度分析器会告诉你：这个函数很健康，分支少、结构清晰。但如果你问三个问题——

1. `g_state` 在哪里被修改？
2. `g_err_cnt` 有互斥保护吗？
3. `get_state()` 返回的是当前tick的状态还是上一轮的状态？

你会发现：代码简单，但**耦合关系分散在四个文件、七个任务里**。静态分析给不出答案，因为全局状态本身就是架构问题，不是函数级问题。

## 资源受限环境下的度量观

在PC后端，圈复杂度衡量的是"这个函数有多少条执行路径"。在嵌入式里，真正该关心的是三件事：

**第一，确定性。** 一个ISR里调用 `malloc`，圈复杂度为零，但你的系统可能永远卡在堆碎片上。

**第二，可追踪性。** FreeRTOS任务间的共享数据，耦合度不在代码里，在任务调度表里。你需要的是调用图+数据流图，不是函数级别的复杂度评分。

**第三，预算约束。** Flash 64KB的MCU上，200行的switch-case状态机可能比"优雅"的虚函数表更值得保留。

## 实践建议

给每个模块画一张**数据流向图**，只标三样东西：全局变量、跨任务调用、中断共享资源。如果某个模块有超过5个这样的出口，拆它。比跑100次 `cppcheck --enable=all` 有用。

**一句话总结：圈复杂度量的是"代码有多复杂"，但嵌入式架构要防的是"状态有多分散"。**

> 《代码大全》第2版："模块的内聚性比任何单一度量都更能预测长期维护成本。"

**今日思考题**：打开你项目里那个"看起来很简单"的FreeRTOS任务，找出所有被它读写的跨模块全局变量。如果数量超过5个，这个任务的耦合度是否已经失控？你会如何重新划分边界？
