## 《C语言接口与实现》：信息隐藏是嵌入式系统的呼吸方式

> *"接口是模块对外的全部，实现是模块对自己的全部。二者之间的墙越厚，系统越不容易崩塌。"*
>
> —— Chris Fraser & David R. Haible, *Implementing C Language Interfaces and Data Structures*, §1.3, p. 12（O'Reilly, 1989）

> *"一个良好的接口应该做到：使用者不需要了解实现的任何细节；即使实现完全重写，只要接口不变，使用者就不需要修改一行代码。"*
>
> —— Chris Fraser & David R. Haible, *Implementing C Language Interfaces and Data Structures*, §4.1, p. 121（O'Reilly, 1989）

### 解读：opaque pointer 在固件中的真实重量

Fraser 和 Haible 在书中反复强调一个朴素到被低估的观点：**接口的本质不是函数签名，而是信息的边界**。

在嵌入式固件中，这个观点不是学术偏好，而是生死攸关的物理约束。

想象你正在为一个电机控制项目编写驱动。一个 naive 的设计可能是这样的：

```c
typedef struct {
    uint32_t freq;
    uint32_t duty;
    uint32_t period;
} pwm_config_t;

void pwm_init(pwm_config_t *cfg);
void pwm_set_duty(uint32_t val);
```

看起来清晰，对吧？但 `pwm_config_t` 暴露了三个字段——`freq`、`duty`、`period`。这意味着调用方可以随意修改它们。如果调用方把 `duty` 设成了超过 `period` 的值呢？如果两个模块同时修改 `freq` 呢？

更关键的是：`pwm_config_t` 的大小直接写进了每个调用方的栈帧。假设你的 PWM 驱动后来增加了 5 个内部状态字段，所有包含这个头文件的模块，栈深度都会膨胀。在资源紧张的 Cortex-M0 上，这可能就是栈溢出的最后一根稻草。

**Opaque pointer 的解药**：

```c
/* pwm.h */
typedef struct pwm_handle pwm_handle_t;
pwm_handle_t *pwm_open(uint32_t channel);
void   pwm_close(pwm_handle_t *h);
void   pwm_set_duty(pwm_handle_t *h, uint32_t val);
```

```c
/* pwm.c */
struct pwm_handle {
    volatile uint32_t *regs;
    uint32_t           freq;
    uint32_t           duty;
    uint32_t           period;
    uint8_t            state;
    /* 未来新增的字段，调用方永远看不到 */
};
```

调用方连 `struct pwm_handle` 的存在都看不到——它只是一个不透明的句柄。`sizeof(pwm_handle_t)` 对调用方而言是零开销。

这在后端开发中可能显得小题大做（内存足够挥霍）。但在嵌入式系统中，每一次 `#include` 都在消耗 Flash 和 RAM。opaque pointer 不仅是封装，更是**编译时依赖的最小化**。

Fraser 和 Haible 给出的经验法则值得刻在开发板上：**如果一个结构体的定义被多个源文件包含，它就应该被拆成接口层和实现层**。这不是过度设计——这是给未来的自己留余地。
